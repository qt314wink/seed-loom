# Document 1 — Master System Brief

Status: v0.1 draft  
System: Visual Ontology Compiler  
Root standard: Document 0 — Universal Entity, Parameter, Relationship, and Transformation Schema

## 1. Mission

The Visual Ontology Compiler, abbreviated VOC, converts visual material into reusable intelligence.

It takes images, materials, patterns, textures, craft processes, art styles, cultural references, frontend interface ideas, prompt ideas, and code implementation ideas, then converts them into searchable taxonomy, prompt templates, design tokens, material definitions, pattern recipes, SVG/CSS/WebGL instructions, QA rules, knowledge graph nodes, governed generative variations, and reusable frontend systems.

Short operating loop:

Observe visual evidence. Define it canonically. Constrain it intelligently. Compile it into prompts and code. Validate it. Store it. Allow governed evolution.

## 2. What VOC is

VOC is a material-aware, craft-aware, renderer-aware, prompt-aware, and QA-governed visual intelligence system.

It is designed to make visual language interoperable across:

- image analysis
- AI prompt generation
- frontend design systems
- CSS/SVG/WebGL rendering
- design tokens
- Figma and Framer handoff
- pattern libraries
- craft and cultural provenance
- knowledge graph storage
- productized asset packs

## 3. What VOC is not

VOC is not a random style generator.

VOC is not a pile of tags.

VOC is not a single-model prompt library.

VOC is not a culture-labeling engine.

VOC is not a renderer-specific material library.

VOC is not allowed to treat vague aesthetic labels as stronger than evidence, material logic, craft logic, or QA.

## 4. Core object

The core object is the Visual Construct.

A Visual Construct can be an image, material, texture, pattern, UI component, prompt, shader recipe, craft process, generated variation, rejected pathway, or product export.

Every Visual Construct inherits the universal model from Document 0:

- identity
- parameters
- relationships
- transformations
- constraints
- QA
- decision memory

## 7. Quality principles

Clarity: every term needs a definition.

Interoperability: every construct should be able to become a prompt, token, code direction, pattern, or graph node.

Evidence first: observation must precede interpretation.

Constraint before mutation: the system defines preserve and avoid rules before generating variants.

Renderer neutrality: SVG, CSS, WebGL, Figma, and AI prompts are compile targets, not the source of truth.

Cultural specificity: use documented process, tool, region, or tradition when known. Avoid vague cultural labels.

Rejected knowledge remains useful: failed variants are archived as dormant, rejected, experimental, or deprecated records.

## 8. Decisions and alternatives

Chosen system spine: universal entity model plus specialized Visual Construct schema.

Reason: this preserves interoperability across visual, semantic, prompt, code, and graph outputs.

Alternative one: build separate systems for images, materials, prompts, and components. Not chosen because it creates translation friction.

Alternative two: start with a large material encyclopedia. Not chosen because vocabulary without operating grammar will sprawl.

Alternative three: start with a frontend app. Not chosen because the schema and QA logic must stabilize first.

Earmarked pathways:

- SVG-first lightweight renderer pack
- Figma-first design-system pack
- prompt-first creator workflow
- knowledge-graph-first research workflow
- wild-lab mode for high-novelty mutations

## 9. Definition of v0.1 success

VOC v0.1 succeeds when it can process a tiny test set and produce:

- stable entity IDs
- canonical parameters
- typed relationships
- positive and negative constraints
- prompt output
- code or token direction
- QA scores
- decision records
- one approved mutation
- one rejected or earmarked mutation

The first v0.1 test set should include five images, three materials, three patterns, one UI component, one generated variation, and one rejected variation.
